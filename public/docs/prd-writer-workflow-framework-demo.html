0. 输入需求

原始想法：我想把一个重复工作做成 AI 工作流，但不知道怎么拆需求、怎么定范围、怎么验收。

PRD Writer Skill 会先做概念对齐，不直接生成一大篇看似完整但不可执行的文档。

1. 概念版对齐

维度结论
一句话定义把重复工作拆成可复用步骤、输入输出和验收标准的工作流框架工具
目标用户个人创作者、产品经理、运营、需要沉淀流程的小团队
核心场景把一次性聊天结果沉淀为可复用 SOP / PRD / Demo 路径
本版不做不做账号系统、不做复杂权限、不做自动计费

2. MVP 范围

模块解决问题验收标准
需求诊断避免一上来写错方向能输出用户、场景、目标、边界和待确认项
工作流拆解把模糊流程拆成阶段、输入、动作、输出能形成阶段表和责任分工
交付检查避免只有文档没有落地能列出成功标志、风险和下一步动作

3. 工作流设计

阶段输入处理输出
D0 资料盘点想法、旧文档、项目文件提炼事实和缺口事实清单
D1 概念对齐事实清单和用户补充确定 MVP 范围概念版 PRD
D2 标准 PRD确认后的范围补流程、数据、验收、风险标准 PRD
D3 网页演示PRD Markdown生成皮玺玉风格 HTML可打开 PRD 网页
D4 复盘迭代使用反馈和新进度更新文档和入口新版 PRD / SOP

4. 验收标准

  • 读者 30 秒内能理解这个工作流解决什么问题。
  • MVP 范围不超过 3 个核心模块。
  • 每个模块都有输入、动作、输出和成功标志。
  • 明确本版不做内容，避免范围失控。
  • PRD 网页可直接打开，无需额外构建。
  • 后续项目能复用同样框架。

5. 风险与待确认

风险影响处理方式
需求太抽象PRD 看似完整但不可执行先做概念对齐再进入标准 PRD
工作流过重用户不愿照着执行MVP 只保留 3 个核心模块
指标编造复盘不可信未知数据标记待确认或待统计
工具依赖变化后续无法复现记录默认工具、版本和兜底方式
